# 项目重命名指南

> 基于 GitHub Template Repository + `chet-rename` 脚本，一键将 `Chet.Admin` 重命名为任意项目名，快速构建新项目。

## 一、方案概述

### 核心思路

1. 将 `Chet.Admin` 仓库在 GitHub 设置为 **Template Repository**
2. 用户通过 "Use this template" 一键复制全量代码到新仓库
3. 在新仓库本地运行 `chet-rename` 脚本，交互式完成所有命名替换
4. 脚本自动处理 12 种命名风格转换，并跑 `dotnet build` 校验

### 为什么不用"下载 zip + 全局替换"

| 问题 | 简单替换 | chet-rename 脚本 |
| ---- | -------- | ----------------- |
| 命名风格多样 | 单一字符串无法覆盖 | 12 种风格映射表 |
| .slnx 结构 | 需手动处理 Path | 自动替换路径字符串 |
| GUID 风险 | 易错 | slnx 零 GUID，无此风险 |
| 误伤上游代码 | 高 | 跳过 `@vben/`、`tools/` 等 |
| 编译校验 | 无 | 自动 `dotnet build` |

### 命名风格映射表

脚本自动从用户输入的 PascalCase 名字（如 `MyApp.Admin`）派生所有风格：

| 风格 | 派生规则 | 原值示例 | 新值示例 | 用途 |
| ---- | -------- | -------- | -------- | ---- |
| PascalCase | 用户输入 | `Chet.Admin` | `MyApp.Admin` | namespace、csproj、文件夹、slnx Path |
| Display | `.` → 空格 | `Chet Admin` | `MyApp Admin` | VITE_APP_TITLE、companyName |
| Kebab | 小写 + `.`→`-` | `chet-admin` | `my-app` | VITE_APP_NAMESPACE |
| RedisPrefix | 去 `.` 拼接 + `:` | `ChetAdmin:` | `MyAppAdmin:` | Redis InstanceName |
| DbFile | PascalCase + `.db` | `Chet.Admin.db` | `MyApp.Admin.db` | 连接字符串 |
| GitHubUrl | 用户输入 | `qiect/Chet.Admin` | `user/MyApp.Admin` | basic.vue 链接 |

## 二、GitHub 配置 Template Repository

### 步骤 1：设置 Template Repository

1. 浏览器打开 `https://github.com/qiect/Chet.Admin`
2. 点击 **Settings**（仓库设置，不是账号设置）
3. 滚动到 **"Template repository"** 选项
4. 勾选复选框
5. 页面顶部会自动保存

设置完成后，仓库页面会有一个绿色 **"Template"** 徽章。

### 步骤 2：用 Template 创建新仓库

1. 在 `qiect/Chet.Admin` 仓库主页点绿色按钮 **"Use this template"** → **"Create a new repository"**
2. 填写：
   - **Owner**: 你的账号或组织（如 `mycompany`）
   - **Repository name**: `MyApp.Admin`（或任意名字）
   - **Description**: 可选
   - **Visibility**: Public / Private
3. 点击 **"Create repository from template"**

GitHub 会把全部代码（含 git 历史）复制到新仓库。新仓库与原仓库无关联。

## 三、本地测试完整步骤

### 前置准备

确保本地已安装：

- Node.js >= 18
- .NET SDK 10
- pnpm（前端用）
- git

### 步骤 1：Clone 新仓库到本地

```bash
cd e:\Project
git clone https://github.com/mycompany/MyApp.Admin.git
cd MyApp.Admin
```

> 注意：clone 后的目录结构是 `e:\Project\MyApp.Admin\tools\chet-rename\`，脚本已随 Template 复制过来。

### 步骤 2：先验证原项目能编译

```bash
dotnet build Chet.Admin.Api\Chet.Admin.slnx
```

应看到 `已成功生成` + `0 个错误`。这步确认 clone 的代码是好的。

### 步骤 3：dry-run 预览

```bash
node tools\chet-rename\bin.mjs --dry-run --name MyApp.Admin --github mycompany/MyApp.Admin
```

检查输出：

- 映射预览表是否正确
- 文件重命名列表是否合理（应该是 14 个 .csproj + .slnx）
- 文件夹重命名列表是否合理（19 个 `Chet.Admin.xxx` 目录）
- 不会有任何文件被修改

### 步骤 4：实际执行重命名

```bash
node tools\chet-rename\bin.mjs --name MyApp.Admin --github mycompany/MyApp.Admin
```

交互式流程会问：

```
? 新项目名称 (PascalCase，可含点，如 MyApp.Admin): MyApp.Admin
? GitHub 仓库地址 (默认: mycompany/MyApp.Admin): mycompany/MyApp.Admin

  项目名 (PascalCase)    : MyApp.Admin
  展示名                 : MyApp Admin
  前端 namespace         : myapp-admin
  Redis 前缀             : MyAppAdmin:
  GitHub 仓库            : mycompany/MyApp.Admin

? 确认以上信息？(y/N): y
```

脚本会依次：

1. 替换 188 个文件内容（923 处匹配）
2. 重命名 14 个文件（.csproj + .slnx）
3. 重命名 19 个文件夹（叶子优先）
4. 跑 `dotnet build` 校验

### 步骤 5：人工校验关键文件

```bash
# 看一眼 slnx，路径应该都变成 MyApp.Admin
type MyApp.Admin.Api\MyApp.Admin.slnx

# 看一眼 appsettings.json，连接字符串 + Redis 前缀应已替换
type MyApp.Admin.Api\MyApp.Admin.Api\appsettings.json

# 看一眼前端 .env
type MyApp.Admin.Web\apps\web-antd\.env
```

预期结果：

```xml
<!-- slnx -->
<Project Path="MyApp.Admin.Api/MyApp.Admin.Api.csproj" />
```

```json
// appsettings.json
"DefaultConnection": "Data Source=MyApp.Admin.db"
"InstanceName": "MyAppAdmin:"
"Issuer": "MyApp.Admin"
```

```bash
# .env
VITE_APP_TITLE=MyApp Admin
VITE_APP_NAMESPACE=myapp-admin
```

### 步骤 6：后端编译验证

```bash
dotnet build MyApp.Admin.Api\MyApp.Admin.slnx
```

应看到 `已成功生成` + `0 个错误`。

### 步骤 7：后端启动验证

```bash
cd MyApp.Admin.Api\MyApp.Admin.Api
dotnet run
```

浏览器访问 `https://localhost:5001/swagger` 或 `http://localhost:5000/swagger`，应能看到 Swagger 标题已变成 `MyApp.Admin`。

### 步骤 8：前端启动验证

另开一个终端：

```bash
cd e:\Project\MyApp.Admin\MyApp.Admin.Web
pnpm install
pnpm dev:antd
```

打开 `http://localhost:5666/`，应看到：

- 浏览器标签标题：`MyApp Admin`
- 登录页 Logo 旁文字：`MyApp Admin`
- 登录后 dashboard 欢迎语：`欢迎回到 MyApp Admin 管理系统`

用 `admin@example.com` / `Admin@123` 登录验证功能正常。

### 步骤 9：提交并推送到 GitHub

```bash
cd e:\Project\MyApp.Admin
git add -A
git commit -m "Rename to MyApp.Admin"
git push origin main
```

## 四、回滚方案

如果任何步骤出错，一键回滚：

```bash
cd e:\Project\MyApp.Admin
git checkout .
git clean -fd
```

这会把所有改动恢复到 clone 时的状态。然后可以调整脚本映射表后重试。

## 五、常见问题排查

| 现象 | 原因 | 解决 |
| ---- | ---- | ---- |
| `dotnet build` 报找不到 csproj | 文件夹重命名顺序错误 | 检查 `collectDirRenames` 是否叶子优先 |
| 前端启动白屏 | `.env` 没替换到 | 确认 `VITE_APP_NAMESPACE` 已改为 `myapp-admin` |
| Swagger 标题没变 | 缓存 | 删 `bin/` `obj/` 重新 `dotnet build` |
| 登录失败 401 | JWT Issuer/Audience 替换后不匹配 | 后端改了 Issuer，前端请求无需改动（前端只带 token） |
| `tools/chet-rename` 自身被改坏 | 跳过规则没生效 | 确认 `SKIP_DIRS` 含 `'tools'` |

## 六、测试用例建议

至少跑 3 种命名验证映射表鲁棒性：

| 测试用例 | 命令 | 验证点 |
| -------- | ---- | ------ |
| 单段名 | `--name MyApp` | 无点的 namespace 是否合法 |
| 两段名 | `--name MyApp.Admin` | 标准场景，点号处理 |
| 三段名 | `--name Company.Product.Admin` | 多段 namespace + Redis 前缀拼接 |

每种都先 `--dry-run` 看映射表，确认无误再实际执行。

## 七、chet-rename 脚本说明

### 项目结构

```
tools/chet-rename/
├── package.json          # 零依赖，Node >= 18
├── bin.mjs               # CLI 入口（交互式 + 命令行参数）
└── src/
    ├── names.js          # 命名风格派生 + 12 项替换映射表
    ├── fs.js             # 扫描 + 内容替换 + 文件/文件夹重命名
    ├── validate.js       # dotnet build 校验
    └── log.js            # 彩色日志 + dry-run 报告
```

### 使用方式

```bash
# 交互式
node tools/chet-rename/bin.mjs

# 非交互式 + dry-run 预览
node tools/chet-rename/bin.mjs --dry-run --name MyApp.Admin --github user/MyApp.Admin

# 实际执行
node tools/chet-rename/bin.mjs --name MyApp.Admin --github user/MyApp.Admin
```

### 安全机制

- `tools/` 目录排除（避免脚本自毁）
- `node_modules`/`bin`/`obj`/`.git` 等构建产物排除
- dry-run 模式只预览不写入
- 文件夹重命名叶子优先，避免路径失效
- 替换后自动 `dotnet build` 校验

### 执行顺序

脚本内部执行顺序（避免路径失效）：

1. 扫描所有待处理文件
2. 替换文件内容（此时路径还没变，引用不会断）
3. 重命名文件（.csproj / .db / .slnx 等）
4. 重命名文件夹（从最深的叶子目录向根目录逐层改）

## 八、与 slnx 的协同优势

项目解决方案文件已从 `.sln` 迁移到 `.slnx` 格式，对重命名脚本的关键意义：

| 场景 | 旧 .sln 方案 | 新 .slnx 方案 |
| ---- | ------------ | ------------- |
| 重命名项目 | 改项目名+路径+GUID+84行配置，4处需同步 | 改 1 个 `Path` 字符串 |
| GUID 冲突风险 | 高（手填易错） | 0（无 GUID） |
| diff 可读性 | 几乎不可读（GUID 散布） | 清晰 XML diff |
| 用户手动微调 | 几乎不可能 | 直接编辑 XML |

slnx 让重命名步骤从"高风险操作"降级为"纯字符串替换"，这是迁移到 slnx 的最大收益。
